# CLAUDE.md

Guidance for Claude Code when working in this repository.

## Project

A pull request dashboard. A Go HTTP API polls GitHub for PR data and serves it to a React/Vite frontend. Postgres for storage, Docker Compose for local orchestration.

## Layout

- `api/` Go service (`dashboard serve`) on `:8080`
  - `cmd/dashboard/` entrypoint and CLI flags (`urfave/cli/v3`)
  - `pkg/auth/` trusted-header user-identity middleware + security headers
  - `pkg/dashboard/` wiring shared between `serve` and the e2e harness
  - `pkg/github/` GitHub client, per-repo poller, ingester
  - `pkg/pullrequest/` domain model, store, service, HTTP handler
  - `pkg/httpserver/` chi-based HTTP server and middleware
  - `pkg/postgres/` pgx pool helpers
  - `pkg/logging/` zerolog setup
  - `migrations/` goose-style SQL migrations (run automatically on boot)
- `app/` React 19 + Vite frontend (JSX, CSS modules, no TypeScript)
  - `src/api/` fetch wrapper and `useApi` hook
  - `src/screens/`, `src/components/`, `src/styles/`
- `e2e/` separate Go module for end-to-end tests + screenshot capture + dev iteration; never imported by the api binary
  - `scenarios/` `make test-e2e` test cases asserting on the rendered DOM
  - `screenshots/` `make screenshots` test cases that write PNGs as a side effect
  - `cmd/dev-e2e/` long-running stack used by `make dev-e2e` for HMR-driven frontend iteration
  - `internal/e2e/` harness core (Stack + Boot for binaries, Harness + Start for tests, chromedp Browser)
  - `internal/githubtest/` fake GitHub server (REST + GraphQL)
  - `internal/pgtest/` Postgres testcontainers helpers
  - `internal/scenarios/` shared seed data (`scenarios.Demo`) consumed by both screenshots and dev-e2e
- `docs/screenshots/` canonical UI captures regenerated by `make screenshots`
- `flake.nix` the single source of truth: pinned toolchain, builds (`packages`), checks, apps, and the dev shell. CI and `make` both call these outputs.
- `Makefile` forwarding-only aliases for the flake outputs (no logic; `make lint` == the `nix` command CI runs)
- `docker-compose.yml` example stack (Postgres + api + oauth2-proxy) running a pinned, published image

## Workflow rules

**Open a pull request at the end of every task. Don't push directly to `main`.**

1. **Branch.** Work on a feature branch.
2. **Commit.** Follow [Conventional Commits](https://www.conventionalcommits.org/en/v1.0.0/): `<type>(<scope>): <imperative subject>`.
   - Types: `feat`, `fix`, `chore`, `docs`, `refactor`, `test`, `ci`, `build`, `perf`. Append `!` after the type or scope to flag a breaking change.
   - Scopes: `api`, `app`, `ci`, `docker`, `deps`, `docs`. Use the most specific scope that fits; omit it only when the change is genuinely cross-cutting.
3. **Describe the why** in the commit body, not just the what.

Lint and unit tests are flake checks: run them locally with `make lint` and `make test-unit`, or `make check` for the full suite. The slower `make test-e2e` boots Docker + Chrome, so run it manually or rely on CI. CI runs the same flake outputs on every PR via `.github/workflows/ci.yml`: the `check` job runs `make check` (a single `nix-fast-build .#checks` over lint, unit tests, and the image), and a separate `e2e` job runs `make test-e2e`. There is no task matrix to keep in sync, because CI invokes the same flake outputs `make` does; add a check by adding it to `checks` in `flake.nix` and it runs everywhere automatically. A `gate` job is the single required status check.

### Screenshots for visual changes

Any PR that touches CSS, layout, copy, icons, or new UI should include a screenshot from the running app. Capture canonical screenshots through the e2e harness:

```bash
make screenshots                      # -> docs/screenshots/
OUT=path/to/dir make screenshots      # -> a custom dir
```

The suite in `e2e/screenshots/` seeds a scenario via the `scenarios` helpers, drives a chromedp tab into the target state, and writes PNGs. It serves the Nix-built `web` bundle (the same one that ships in the image), so the docs cannot drift from what is deployed. To capture a new state, add a test there. Requires Docker and Chrome; the flake pins Chromium on Linux (`CHROME_PATH`), and on macOS the harness auto-detects a `/Applications` Chrome.

For interactive frontend iteration, `make dev-e2e` runs the same stack long-lived behind Vite with HMR on live working-tree frontend source. The dev binary stamps `X-Forwarded-User` server-side, so there is no sign-in step locally.

## Commands

The flake is the single source of truth; `make` targets are forwarding-only aliases for the same flake outputs CI runs (run `nix` directly if you prefer). `nix develop` drops you into the dev shell with every pinned tool (Go, Node, golangci-lint, oauth2-proxy, Chromium, jq, nix-fast-build).

```bash
make help                # list targets
make lint                # Go, JS, and shell linters (builds the lint-* flake checks)
make test                # unit + e2e tests (test-unit then test-e2e)
make test-unit           # unit tests, api/pkg + api/cmd (builds the test-unit flake check)
make test-e2e            # full e2e suite, Docker + Chrome required (nix run .#test-e2e)
make dev-e2e             # long-lived harness + Vite HMR proxy
make screenshots         # writes canonical PNGs to docs/screenshots/ (OUT=dir overrides)
make check               # the full flake check suite (nix-fast-build .#checks)
make image               # build the x86_64-linux OCI image and docker load it
make tidy                # tidy go.mod + regenerate the gomod2nix lockfile
nix develop -c <cmd>     # run a command in the dev shell
cd api && go run ./cmd/dashboard serve   # run the api directly
docker compose up        # example stack (pulls a pinned published image)
```

Required inputs for `dashboard serve`: `DASHBOARD_DATABASE_URL`, `DASHBOARD_GITHUB_TOKEN`, and `--config` / `DASHBOARD_CONFIG` pointing at the YAML config file. Authentication is a trusted-header proxy's job; the dashboard reads the signed-in user from `X-Forwarded-User`.

## Conventions

### Go (api/)

- Stdlib first; approved deps: `chi`, `pgx/v5`, `golang-migrate`, `go-github/v85`, `urfave/cli/v3`, `zerolog`, `yaml.v3`, `testify`.
- Migrations are append-only; add a new pair `YYYYMMDDHHMMSS_name.up.sql` / `.down.sql` rather than editing existing files.
- HTTP routes are mounted under `/api`; the frontend SPA is served for everything else.
- When you add a new per-PR field or related table populated by the ingester, bump `IngestVersion` (see below).

### Frontend (app/)

- React 19, JSX (no TypeScript), CSS Modules (`*.module.css`), design tokens in `src/styles/tokens.css`.
- Run `npm run lint` before committing; don't introduce new ESLint warnings.
- Keep components small and colocate styles (`Foo.jsx` + `Foo.module.css`).

## Things to keep in mind

- The poller runs one ticker per repo so a slow repo cannot block a fast one. Preserve this isolation if you touch `pkg/github/poller.go`.
- No webhook setup is required by design; everything works with a personal access token (`repo` scope). Don't introduce features that require admin/webhook access on the target repos.
- Repositories and review rules are **per-user**, stored in the database and edited from the in-app Settings screens, not in the YAML config. The `--config` file now carries only server-level operational settings (the default poll interval). The relevant pieces:
  - `user_repos`, `user_settings`, `user_reviewer_overrides` tables (migration `20260626120000_per_user_settings`).
  - `pkg/pullrequest/UserStore` persists them; `SettingsHandler` serves `/api/settings/repos`, `/api/settings/rules`, and the per-user `/api/config`.
  - `PostgresService.List` loads the viewer's observed repos + settings, snapshots only those repos, and applies the viewer's `Rules`. Built-in defaults for a user with no saved row live in `pkg/pullrequest/usersettings.go`.
  - The poller services the **union** of every user's observed repos (`UserStore.DistinctRepos`), reconciling that set dynamically: a repo keeps polling while anyone observes it and stops once the last subscriber removes it. Preserve the one-ticker-per-repo isolation.
  - Don't reintroduce a global YAML list of repos or review-policy labels.
- Authentication is a trusted-header proxy's job. The `TrustedHeader` middleware reads `X-Forwarded-User` and injects `auth.User` into request context. The server PAT (`DASHBOARD_GITHUB_TOKEN`) does all GitHub API work (polling, ingest, on-demand poll) and is never tied to a specific user. Don't reintroduce in-process OAuth handlers, session storage, or org-membership checks in the dashboard itself.
- The e2e harness simulates the proxy by stamping `X-Forwarded-User` on every request reaching the dashboard handler (see `e2e/internal/e2e/stack.go`'s `injectAuthHeader`). Tests configure the user via `e2e.WithViewer`.
- `pull_request_views` rows are created lazily by `MarkViewed`, keyed `(user_login, pr_github_id)`. There is no backfill and no per-user baseline; a new user sees every "mine"/"reviewed" PR as unread until they mark them. Don't reintroduce per-user backfills or a global single-row view cache.

### Bumping `IngestVersion`

The poller advances a per-repo cursor and only refetches PRs touched on GitHub since the last sync. That makes it efficient but blind to schema changes: when we add a new ingestable per-PR field, pre-existing rows that haven't been touched won't get the new field populated. Every PR row carries an `ingest_version` column and the ingester writes the constant `IngestVersion` (defined in `api/pkg/github/ingest_version.go`). At the end of each poll tick, the poller refetches up to `staleBudgetPerTick` active PRs (open, or merged within 7 days) whose stored version is below current. The cap drip-feeds the backfill so a version bump can't burst-spend GitHub quota right after a deploy, and a degraded GitHub can't stall a repo's catch-up (failures retry on the next tick).

When you add a new per-PR field or related table populated from `handlePR` or `handleReview`, you **must**:

1. Increment `IngestVersion` in `api/pkg/github/ingest_version.go` by exactly 1.
2. Add a one-line entry to the version history comment in that file describing the bump and the migration timestamp it goes with.

Do **not** bump for changes that only affect how existing fields are read or computed (a new SQL view, a derived field in `view.go`, a rules-layer change). Bump only when the ingest pipeline starts writing data that pre-existing rows don't already have.
